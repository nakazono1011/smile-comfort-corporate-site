Bright Data の Residential と ISP プロキシ、どちらを選ぶべきか。結論は、**コストと速度を取るなら ISP を第一選択、超難関サイトや厳しい IP 多様性が必要なときだけ Residential** が現実的です。本稿では、両者の構造的違い、2026 年時点の比較表、運用してきた切り替え基準と段階的フォールバック設計を整理し、無駄なコストをかけずに成功率を維持する判断材料を示します。

## Residential と ISP の構造的な違い

両者は「住宅用 IP を経由する」という点では同じですが、出口の物理的位置とアドレスの取得経路が異なります。

### Residential プロキシは家庭回線が出口

Residential プロキシは、Bright Data の SDK に同意した一般家庭ユーザーの PC やスマートフォンの回線を経由します。出口の IP は住宅 ISP が割り当てた本物の住宅 IP で、ASN・GeoIP・行動パターンのいずれから見ても家庭ユーザーと区別がつきません。グローバルで 7,200 万を超える IP プールが利用可能で、都市単位やキャリア単位での絞り込みもできます[^1]。

一方、回線品質は提供元の環境に依存します。家庭回線特有の遅延・不安定さがあり、長時間同一 IP を維持しにくい仕様です。

### ISP プロキシはデータセンターから出る住宅 IP

ISP プロキシは、ISP が住宅向けに割り当てた IP アドレス枠を Bright Data がデータセンターのサーバーに紐付けて運用するハイブリッド型です。外部から見える ASN は住宅 ISP のまま、サーバー側で動いているため帯域は安定し、レイテンシも低く抑えられます。長時間同一 IP でセッションを維持できる「Static Residential」として扱われます。

価格モニタリング基盤の具体的な構築は [Bright Data Residential プロキシで価格モニタリング基盤を構築する方法](/media/brightdata-residential-price-monitoring) で整理しているため、用途のイメージが掴みにくい方は合わせて確認してください。

## 検知耐性・速度・コストで見る比較表

意思決定で見るべき軸は、検知耐性・速度・コスト・固定性の 4 つです。Bright Data 公式の製品比較ページ[^2]と、弊社の運用実績から得られた感覚値をベースに整理します。

| 評価軸 | Residential | ISP | 用途別の勝者 |
|---|---|---|---|
| 検知耐性 (Stealth) | 最高クラス | 高いが Residential より一段下 | Residential |
| 速度・スループット | 中 (家庭回線依存) | 高 (DC グレード) | ISP |
| 接続安定性 | 中 (ユーザー離脱あり) | 高 (24/7 稼働) | ISP |
| GB 単価 | 高 (相対的に) | Residential より明確に安い | ISP |
| 固定 IP セッション | 短時間に限定 | 長時間維持しやすい | ISP |
| 都市・キャリア単位ターゲティング | きめ細かく可能 | 利用可能 | Residential |
| ボリュームディスカウント適用後の最終単価感 | 中量帯から効く | 低量帯から効く | ISP |

ボリュームディスカウントや年間契約の圧縮余地は [Bright Data 料金プラン早見表 2026](/media/bright-data-pricing-2026) にまとめています。コスト試算時に合わせて確認してください。

### Bright Data の ISP は IP の鮮度がとくに高い

Bright Data の ISP は、他社の Static Residential と比較して IP のクリーン度 (過去のスパム履歴の少なさ) が高い評価のプロダクトです。X 上でも「ISP をメイン、詰まったときだけ Residential」という運用例が複数報告されています。

> 「ISP をメインのレーンに据えて、ISP では突破できない案件だけ Residential にフォールバックさせている。Bright Data の ISP は IP がクリーンで運用負荷も低い」(原文: ISP as the primary lane, fall back to Residential only when ISP can't handle it. Bright Data's ISP pool stays clean.)

## 実践的な使い分けの判断フロー

「どちらを使うべきか」を毎回手探りで決めるのは非効率です。下の 3 ステップで過剰なコストを払わずに進められます。

### Step 1. まずは ISP で全件試す

新規プロジェクトで対象サイトのプロキシ要件が分からない場合、最初の試行は ISP で揃えるのが定石です。Residential より GB 単価が安く、速度が安定しているため、初期の試行錯誤コストを抑えられます。

1. 対象サイトのサンプル URL を 100〜500 件用意する
2. Bright Data の ISP Zone を作成し、IP プールを 100〜200 個割り当てる
3. リクエスト成功率・CAPTCHA 出現率・レスポンス時間を計測する

成功率 90% 以上、CAPTCHA 出現率 5% 未満なら ISP のままで問題ありません。

### Step 2. 失敗時のみ Residential にフォールバック

ISP の失敗率が許容範囲 (例: 10%) を超えるエンドポイントが見つかったら、その分だけを Residential に流す 2 段構成に切り替えます。

1. ISP で 1 回目を取得
2. HTTP 403/429 や CAPTCHA HTML を検知したら同じ URL を Residential Zone で再取得
3. それでも失敗した場合は Web Unlocker か Mobile に切り替え、もしくはスキップ

Bright Data の Proxy Manager か自前のリクエストレイヤーで実装します。月間トラフィックの 70〜80% を ISP、残り 20〜30% を Residential という配分にすると、Residential 単独と比べてプロキシコストを 30〜50% 削減できます。

### Step 3. それでも厳しいときだけ Mobile / Web Unlocker

ISP と Residential の組み合わせでも成功率が伸びない場合、ターゲットサイトの bot 検知が非常に強い可能性があります。Mobile Proxy は 4G/5G キャリア IP 経由で高い検知耐性を持つ反面、GB 単価は Residential より高くなります。Web Unlocker は CAPTCHA や JavaScript チャレンジを内部処理するため自前のリトライ実装を簡略化できます。

## 運用上の注意点とコンプライアンス

両者ともプロキシとしての挙動は近いものの、運用上は別物として扱うべき点があります。

### IP プールの管理とローテーション

ISP は静的 IP のため、ブロックされた IP は休ませるか別 IP に切り替える運用が必要です。Bright Data のダッシュボードから IP リスト管理とローテーション設定が可能で、用途ごとに Zone を分けるとブロック影響を局所化できます。Residential は IP プールが大きく自動ローテーションで概ね問題は出ませんが、セッション維持には `sticky session` の設定を意識してください。

### 利用規約とレート制御

どのプロキシを選ぶ場合でも、対象サイトの利用規約と robots.txt は確認が必要です。Bright Data は KYC を経たユーザーのみが Residential / ISP を利用できる設計で、出口 IP のコンプライアンス面は担保されています[^3]。それでも、スクレイピング側でレート制御や Crawl-Delay の尊重を組み込むことは法務リスクの観点から推奨されます。

> 「プロキシ自体がクリーンでも、対象サイトの ToS と Crawl-Delay を尊重しない運用は法務レビューで必ず止められる。レート制御と robots.txt の確認はスクレイパー側で入れておくべき」(原文: Even with KYC-clean proxies, ignoring ToS and Crawl-Delay will get blocked by legal review. Build rate control on your side.)

エンタープライズ案件で導入する場合、社内法務レビュー段階で「使用する IP の取得経路」が論点になるケースが多いです。Residential の同意取得の仕組みと ISP の ASN 情報は、Bright Data のコンプライアンスドキュメントから資料化できます。

<Callout type="warning" title="注意">
ISP の固定性を活かしてアカウントを長期間維持する用途は、対象サービスの利用規約で「単一ユーザーの真正なアクセス」と明示されている場合に違反となる可能性があります。社内利用方針と合わせて事前に法務確認を行ってください。
</Callout>

## まとめ

Residential と ISP の使い分けは、**「ISP をデフォルトに、失敗分だけ Residential」** で組むのが現実的です。2 段フォールバックを Proxy Manager で構築すれば、Residential 単独と比べてプロキシコストを 30〜50% 圧縮しつつ、エンタープライズグレードの成功率を維持できます。まずは小さな対象サイトで成功率を計測してから本番ロードに乗せてください。
