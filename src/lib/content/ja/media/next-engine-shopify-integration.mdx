ネクストエンジンとShopifyは、公式の無料連携アプリを使えば受注の自動取込・在庫の双方向連動・出荷通知の反映までを一気通貫で自動化できます。ただし、商品コードの紐づけ設計を最初に固めておかないと、在庫ズレや受注の取込漏れといったトラブルが起きやすいのも事実です。本記事では前提条件から設定手順、つまずきやすいポイントの早見表、設定をやり直すためのロールバック手順までを、実運用目線で順に解説します。

## 連携でできること・できないこと（前提条件）

[ネクストエンジン](https://base.next-engine.org/account/?agent_code=MzEzNw)（Next Engine）は、Amazon・楽天・Yahoo!ショッピングなどの受注と在庫を1画面で一元管理するEC向けのOMS（受注管理システム）です。Shopifyも公式の連携アプリを通じてこの管理対象に組み込め、複数モールとShopifyの在庫・受注をまとめて扱えるようになります。この公式連携アプリは単なる在庫連携ツールではなく、複数モールを横断して受注・在庫・出荷をまとめて扱う一元管理OMSの一部として動くのが特徴です。まず、自動化できる範囲と手作業が残る範囲を押さえておくと、あとで設定につまずいたときに原因を切り分けやすくなります。

### 連携に必要な前提条件

連携を始めるにあたって、次の前提を用意しておきます。

- ネクストエンジンの契約アカウント（受注・在庫・商品マスタの各機能が使える状態）
- 有料プランのShopifyストア（API連携を許可できる管理者権限）
- Shopify App Storeで提供される公式の自動連携アプリの導入（アプリ自体は無料[^2]）
- 商品コード（SKU）が整理された商品マスタ
- 両サービスで連携設定を行える管理者アカウント

### 自動化できる範囲と手作業が残る範囲

自動化できる範囲と、手作業が残る範囲は次のとおりです。

| 区分 | 連携で自動化できる | 手作業が残る |
|---|---|---|
| 受注 | 通常注文の自動取込 | サブスク・アプリ経由注文の一部 |
| 在庫 | 双方向の在庫連動 | 複数ロケーションの配分調整 |
| 出荷 | 送り状番号・配送業者の反映 | ネクストエンジン側キャンセルの反映 |
| 商品情報 | （連携対象外） | 商品説明・画像はShopify側で編集 |

アカウント開設やモール接続といった初期設定がまだの場合は、先に [ネクストエンジン初期設定 完全ガイド](/media/next-engine-initial-setup) で土台を整えておくとスムーズです。費用面では、連携に使う公式の自動連携アプリは無料で、料金が発生するのは本体側の月額3,000円〜（税抜・受注200件まで）に、201件目からの受注件数に応じた階層別従量課金（201〜400件は35円/件で、件数が増えるほど逓減）が加わる二層構造です[^1]。プラン別の詳細は [ネクストエンジンの料金プラン徹底解説](/media/next-engine-pricing-2026) にまとめています。

## 連携の設定手順（番号付きチェックリスト）

実際の設定は、公式アプリを追加してから商品コードを紐づけ、在庫・受注・出荷の各連動を有効化する流れで進めます。以下のチェックリストの順に進めると、依存関係の順序で詰まりにくくなります。

1. Shopify App Storeまたはネクストエンジン管理画面から、公式の自動連携アプリを追加する（無料）
2. 連携するShopifyストアを登録し、API接続（アクセストークンの発行・許可）を行う
3. 商品コードを紐づける：ネクストエンジンの商品マスタとShopifyのSKUを、代表商品コードで対応させる
4. 在庫連動を設定する：同期の方向（双方向）と対象ロケーションを指定し、自動同期をONにする
5. 受注取込を設定する：取込対象の注文ステータスと、取込を自動起動するタイミングを決める
6. 出荷通知の連携を設定する：出荷確定時に送り状番号・配送業者をShopifyへ返すよう指定する
7. テスト注文を1件流し、受注取込 → 在庫減算 → 出荷通知までの一連が流れるかを検証する

この一連の処理は、[ネクストエンジン](https://base.next-engine.org/account/?agent_code=MzEzNw)の自動連携アプリがAPI経由で無人実行するため、一度正しく組めば以降は運用の手を離れます。Yahoo!ショッピングなど他モールも基本の流れは同じで、モール固有の認可・注意点は [ネクストエンジン Yahoo!ショッピング連携の設定手順](/media/next-engine-yahoo-shopping-integration) にまとめているので、複数モールを並行して整える際の参考にしてください。

## 商品コード・SKUの紐づけ設計

連携でもっとも事故が起きやすいのが、商品コード（SKU）の紐づけ設計です。ここを曖昧にしたまま在庫連動をONにすると、別商品の在庫が引き当てられたり、在庫数がまったく合わなくなったりします。基本方針は、[ネクストエンジン](https://base.next-engine.org/account/?agent_code=MzEzNw)の商品マスタ側を親（基準）として設計し、ShopifyのSKUをそこに合わせることです。

とくに「代表商品コード」という考え方を理解しておくと設計がぶれません。代表商品コードは、色・サイズなどのバリエーションをまとめる親コードで、在庫を数える単位を明確にする役割を持ちます。

| パターン | 設計のポイント | ありがちな失敗 |
|---|---|---|
| 単品 | Shopify SKU とマスタの商品コードを1対1で対応 | 空白・全角半角の混在でコード不一致 |
| バリエーション | 色・サイズごとに個別SKU、代表商品コードで束ねる | 代表コードだけ登録し在庫単位が曖昧 |
| セット・まとめ売り | 構成品の在庫按分（コンポーネント在庫）で管理 | セット商品を単品扱いにして在庫二重計上 |

セット商品やまとめ売りは、見た目は1商品でも在庫は構成品から引くため、按分設定を忘れると在庫不整合の主因になります。バリエーション商品は、Shopify側のSKUが1つでも欠けると連動対象から漏れるので、商品マスタとSKUの一覧を突き合わせてから連動をONにしましょう。

## つまずきやすいポイントと対処早見表

設定自体は手順どおりでも、運用に入ってから顕在化するトラブルがあります。X（旧Twitter）でも、複数ロケーション環境での在庫ズレやアプリ経由注文の取込漏れに関する声が繰り返し共有されています。よくある症状と原因・対処を先に押さえておきましょう。

| 症状・エラー | 原因 | 対処法 |
|---|---|---|
| アプリ経由・サブスク注文が取り込まれない | 対象の注文種別・ステータスが取込条件の外 | 取込対象ステータスを見直し、サブスクは個別運用ルールで補う |
| 複数ロケーションで在庫がズレる | 同期対象ロケーションが複数で配分が不定 | 基準ロケーションを1つに集約、または連動対象を明示 |
| セット商品の在庫が合わない | 構成品の在庫按分設定の漏れ | セットはコンポーネント在庫で管理し、まとめ売りは別SKU化 |
| Shopify側のキャンセルが反映されない | キャンセルは自動連携の対象外 | 両サービスで手動キャンセルし、在庫を手動調整 |
| 商品ページが更新されない | 商品情報は連携対象外 | 商品説明・画像はShopify側で編集する前提に |

上記のように、取込漏れや在庫のズレは「設定ミス」よりも「取込条件やロケーション設計の想定漏れ」から起きるケースが目立ちます。

在庫不整合の多くはSKU設計に起因するという指摘は、実運用でも当てはまります。連動をONにする前に、商品コードの一覧チェックを済ませておくと事故を大きく減らせます。

## 設定の検証とロールバック手順

本番運用に移す前に、忘れずテスト注文で一連の流れを検証します。検証は「入口（受注）から出口（出荷通知）まで」を1件通すのが基本です。

1. Shopify側でテスト注文を1件作成する
2. ネクストエンジンに受注が取り込まれ、対象SKUの在庫が減算されるか確認する
3. 出荷確定を行い、送り状番号・配送業者がShopify側に反映されるか確認する
4. 在庫数がShopify・ネクストエンジン双方で一致しているか突き合わせる

想定と違う挙動が出た場合は、いったん自動同期をOFFにして連動を止め、商品コードの紐づけと取込条件を見直します。連動を止めれば在庫の自動上書きは発生しないため、落ち着いて原因を切り分けられます。

<Callout type="warning" title="在庫を戻すときの注意">
在庫連動をONのまま商品コードを変更すると、誤った数量で上書きされることがあります。ロールバック（連動の解除・やり直し）を行う前に、現在の在庫数を控えてから作業してください。テスト注文で作った受注データも、本番と混ざらないよう忘れずに削除・キャンセル処理をしておきましょう。
</Callout>

## 弊社の支援と、連携をスムーズに始めるために（まとめ）

### この連携が向いている人・向いていない人

すべてのShopify運営者にこの連携が必要なわけではありません。導入前に、自社が「向いている」側かを確認しておきましょう。

| 向いている | 向いていない |
|---|---|
| 複数モール＋Shopifyを併用し、SKU・在庫管理が煩雑になっている | Shopify単店舗のみで、他モール展開の予定がない |
| 月間受注が一定以上あり、手作業の取込・在庫更新が限界 | 受注件数が極端に少なく、手運用で十分回る |
| 商品マスタを一元管理して他チャネルにも横展開したい | 商品数が少なく、在庫連動の必要性が薄い |

Shopify単店舗で受注も少ない段階なら、Shopify標準の在庫機能や軽量な在庫管理アプリで足りるケースが多く、OMSの導入はオーバースペックになりがちです。まずは自社の受注ボリュームとチャネル数を基準に判断してください。

[ネクストエンジン](https://base.next-engine.org/account/?agent_code=MzEzNw)とShopifyの連携は、公式の無料アプリで受注・在庫・出荷を自動化できる一方、商品コードの設計と複数ロケーションの扱いを最初に固めることが成否を分けます。設定手順そのものよりも、SKU設計と取込条件の想定漏れを先回りで潰しておくことが、運用後のトラブルを減らす近道です。

弊社スマイルコンフォートはネクストエンジンの導入実績があり、初期設定からモール連携・商品マスタの整備・API連携まで一気通貫で支援してきました。月数件〜年商数十億規模のECまで対応してきた経験から、無理のない在庫・受注オペレーションの設計をご提案できます。連携でつまずいたときの切り分けや、既存システムからの商品マスタ移行もあわせてご相談いただけます。
